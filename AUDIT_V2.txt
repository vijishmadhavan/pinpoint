#1  Command injection via name_contains in /list_files (WSL path)
    File: api/files.py:112-114
    Status: FIXED — sanitize shell metacharacters from name_contains
    The name_contains parameter is interpolated into a cmd.exe shell command:
      dir_cmd = f"dir /s /b {win_folder}\\*{name_contains}*"
      subprocess.run(["/mnt/c/.../cmd.exe", "/c", dir_cmd], ...)
    Attacker sends name_contains="foo & del /q C:\*" -> arbitrary command execution.
    The Linux path (line 130) uses array form and is safe.
    Impact: Remote code execution via WhatsApp bot or any API caller.
    Fix: Sanitize name_contains (strip & | ; $ ` etc.) or use os.walk with Python-side filtering.

#2  exec() in run_python with full __builtins__ — no sandboxing
    File: api/transform.py:812-900
    Status: WON'T FIX — by design (power feature). API_SECRET gates access.
            Server-side _check_safe blocks writes to system paths.
    The /run-python endpoint passes __builtins__: __builtins__ to exec().
    Setup code explicitly imports os, sys, shutil. Anyone with API access can
    run arbitrary commands, read/write any file, exfiltrate env vars (GEMINI_API_KEY).
    Impact: Full system compromise if API is reachable (listens on 0.0.0.0:5123).
    Fix: If intentional, ensure API_SECRET is always required and non-empty.
         Better: run in subprocess with restricted permissions or nsjail.

#3  pd.eval(engine="python") allows arbitrary Python execution
    File: api/data.py:435-454
    Status: FIXED — removed engine="python", now uses default numexpr engine
    pd.eval with engine="python" delegates to Python's eval(). The blocklist
    checks for "__", "import", etc. but blocklists are inherently bypassable.
    df.__class__.__bases__[0].__subclasses__() would be caught by "__" check,
    but more creative bypasses may exist.
    Impact: Potential code execution via crafted DataFrame query.
    Fix: Switch to engine="numexpr" (only supports arithmetic, much safer).

#4  Missing _check_safe on ALL face recognition endpoints (8 endpoints)
    File: api/faces.py:72-247
    Status: FIXED — added _check_safe to all 8 endpoints
    These endpoints accept file/folder paths but never call _check_safe:
    - /detect-faces (single image), /crop-face, /find-person,
    - /find-person-by-face, /count-faces (single image), /compare-faces,
    - /remember-face, /recognize-faces
    Only /count-faces with req.paths calls _check_safe.
    Impact: Path traversal — face detection on files in protected directories.
    Fix: Add _check_safe(os.path.abspath(path)) to every endpoint.

#5  Missing _check_safe on ALL photo endpoints (4 endpoints)
    File: api/photos.py:24-88
    Status: FIXED — added _check_safe to all 4 endpoints + rejects/uncategorized folders
    /score-photo, /cull-photos, /group-photos, /suggest-categories
    never call _check_safe on path/folder inputs.
    Impact: Path traversal to score/cull/move photos from protected directories.
    Fix: Add _check_safe calls for req.path, req.folder, req.rejects_folder.

#6  Missing _check_safe on Google endpoints
    File: api/google.py:210-227 (drive-upload), api/google.py:57-71 (gmail-send attachment)
    Status: FIXED — added _check_safe to drive-upload and gmail-send attachment
    /drive-upload takes a local file path and uploads to Google Drive — no _check_safe.
    /gmail-send attachment path is checked for existence but not with _check_safe.
    Impact: Data exfiltration from system directories to Google Drive or via email.
    Fix: Add _check_safe(path) after os.path.abspath.

#7  Missing _check_safe on /ocr single-path mode
    File: api/media.py:337-342
    Status: FIXED — added _check_safe before _ocr_single
    When req.path is used (not req.folder), path goes directly to _ocr_single
    without _check_safe.
    Impact: OCR can read files in protected directories.
    Fix: Add _check_safe(os.path.abspath(req.path)).

#8  Missing _check_safe on /index endpoint
    File: api/core.py:37-121
    Status: FIXED — added _check_safe after os.path.abspath
    /index takes a folder path and indexes all files. Checks os.path.isdir
    but never calls _check_safe.
    Impact: Content from protected directories leaks into search database.
    Fix: Add _check_safe(folder) after os.path.abspath.

#9  write_file / delete_file have no path restrictions on bot side
    File: bot/src/tools.js:1509, 1553-1557
    Status: WON'T FIX — server-side _check_safe already blocks unsafe paths.
            Bot-side validation would be defense-in-depth but API is the gate.
    The LLM can write/delete files at any absolute path. "Ask user first"
    is just an LLM instruction, not enforcement. Prompt injection could bypass.
    Impact: System compromise via prompt injection in indexed documents.
    Fix: Validate paths are under USER_HOME or an allowlist. Server-side
         _check_safe already protects for API-direct calls but the bot
         constructs paths before calling the API.

#10 pdf_to_excel route is broken — tool will ALWAYS fail
    File: bot/src/tools.js:1591
    Status: FIXED — changed q: to p: function with enc() query params
    Route uses q: property which the dispatch code never reads:
      pdf_to_excel: { m: "GET", p: "/pdf-to-excel", q: (a) => ({...}) }
    The dispatch code only reads route.p and route.b. API call goes to
    /pdf-to-excel with no query params -> server error every time.
    Fix: Change to p: (a) => `/pdf-to-excel?path=${enc(a.path)}...`

#11 memoryEnabled is global — shared across all users
    File: bot/index.js:97
    Status: WON'T FIX — by design (single-user tool, shared memory store)
    Single boolean. If user A runs /memory off, memory is disabled for
    user B too. Same for memoryContext (line 98).
    Fix: Make per-chat: Map<chatJid, boolean>.

#12 Recursive startBot() stacks event listeners on reconnect
    File: bot/index.js:1865
    Status: FIXED — clear reminder interval before creating new socket on reconnect
    Each reconnect calls startBot() which calls sock.ev.on() on a NEW socket.
    Old socket's listeners are never removed. global._reminderInterval
    captures stale sock from closure — reminders fail silently after reconnect.
    Fix: Clear old listeners. Store sock in module-level var that reminder
         interval references.

#13 Reminder interval captures stale sock reference
    File: bot/index.js:1801-1842
    Status: FIXED — module-level currentSock updated on reconnect, interval uses currentSock
    After reconnect, new sock is created but interval still uses old one.
    sock.sendMessage() on disconnected socket throws, catch logs but
    doesn't re-queue — reminders are silently lost.
    Fix: Use module-level currentSock variable updated on each reconnect.

#14 DB connection leaks in image_search.py and video_search.py
    File: image_search.py:165,189; video_search.py:137,167
    Status: FIXED — shared module-level connection (created once, reused)
    _get_conn() creates a new sqlite3.Connection on every call, never closes it.
    Under concurrent use (ThreadPoolExecutor in photo_cull.py), many leaked connections.
    Impact: File descriptor exhaustion and SQLite locking contention.
    Fix: Close connections after use, or use module-level connection with lock.

#15 _mem_cache grows without bounds
    File: image_search.py:54
    Status: FIXED — LRU eviction at 10 folders max
    Dict mapping folder_abs to numpy arrays of embeddings. Never evicted.
    Each folder with 1000 images stores ~3MB. Many folders = unbounded growth.
    Fix: Add LRU eviction (limit to 10 most-recent folders).

#16 Zip bomb / symlink traversal in /extract-archive
    File: api/transform.py:724-749
    Status: FIXED — symlink check + 2GB uncompressed size limit
    normpath resolves ".." syntactically but NOT symlinks. Archive with
    symlink pointing outside extraction dir passes normpath check.
    No total uncompressed size limit — zip bomb fills disk.
    Fix: Check for symlinks in archive members. Add total size limit.

#17 _ocr_single doesn't close PIL Image
    File: api/files.py:510-526
    Status: FIXED — try/finally to close both original and preprocessed images
    Image.open(path) at line 515 never closed. _preprocess_image may return
    a new image (original leaks). Repeated OCR calls leak file descriptors.
    Fix: Use with Image.open(path) as img: or explicit close.

#18 _cosine_sim division by zero
    File: face_search.py:329
    Status: FIXED — denom = max(..., 1e-12)
    np.linalg.norm(a) * np.linalg.norm(b) can be zero if either embedding
    is all-zeros (failed face detection). Returns NaN, breaks all comparisons.
    Fix: denom = max(np.linalg.norm(a) * np.linalg.norm(b), 1e-12)

#19 shutil.move overwrites files with duplicate names in cull/group
    File: photo_cull.py:452, 1138
    Status: FIXED — check dest existence, add _1, _2 suffix to avoid collisions
    Two photos from different subfolders with same basename — shutil.move to
    rejects/category folder silently overwrites the first one.
    Impact: DATA LOSS — photos with duplicate filenames get overwritten.
    Fix: Check destination existence, add suffix (_1, _2) to avoid collisions.

#20 compactHistory clears active state mid-conversation
    File: bot/index.js:1174-1176
    Status: FIXED — DB-only reset in compactHistory, no in-memory state cleared
    compactHistory() is fire-and-forget, calls resetSession() which deletes
    actionLedger[chatJid] and sessionCosts[chatJid]. If this runs while
    runGemini() loop is active, LLM loses track of what it did.
    Fix: compactHistory should do DB reset without clearing in-memory state.

#21 Race condition on _cull_jobs/_group_jobs progress dicts
    File: photo_cull.py:219-220, 388-389
    Status: WON'T FIX — CPython GIL makes individual dict key updates atomic.
            Inconsistent reads are possible but harmless (progress polling).
    Progress dict shared between background thread and main thread.
    _cull_lock protects jobs dict but progress keys mutated without lock.
    Fix: Hold lock when mutating progress from background thread.
